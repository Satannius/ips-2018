Hints for Task 2

This document contains hints and guidelines for solving Task 2 of the project task for the 2018 iteration of the course Implementation of Programming Languages at DIKU.

You are not required to follow these hints, but you are strongly encouraged to do so.

This task involves implementing three array combinators (special array-oriented functions): replicate, filter, and scan. We provide full implementations of two other combinators to show you how it can be done: iota and reduce.

Task 2 asks you to make fundamental additions to your Fasto compiler, and write a lot of code. To get you properly started on this large task, this document provides a walkthrough of the existing implementation of iota.

Once you have an idea of how iota was implemented, try implementing replicate — the two constructs are very similar. Once you have implemented replicate, try working your way through map and scan. Of course, you can also choose to completely disregard this help and implement them in the reverse order.

Overview of Fasto’s array combinators

Before you hack away, here is a quick overview of the six array constructors and combinators in Fasto, three of which you need to implement. Remember to also read the group project PDF file, which provides more detail.

iota (existing)

Type: int → [int]

Result for iota(n): [0, 1, …​, n - 2, n - 1]

Example: iota(5) should return [0, 1, 2, 3, 4].

replicate (yours!)

Type: (int * a) → [a]

Result for replicate(n, x): [x, x, …​, x, x] (n times)

Example: replicate(4, true) should return [true, true, true, true].

map (existing)

Type: ((a → b) * [a]) → [b]

This has the same semantics as the List.map function in F#.

Result for map(fn int (x:int) ⇒ x+1, iota(n)): [1, 2, …​, n]

filter (yours!)

Type: ((a → bool) * [a]) → [a]

This has the same semantics as the List.map function in F#.

Notation: we call a predicate a function that takes one argument and results in a boolean.

Filter receives a predicate argument and an array, such as the type of the predicate’s parameter is the same with the array element type. Filter results into a new array whose elements are the ones from the input array who succeed under the predicate. The resulting elements should appear in the same order as in the original array. The length of the resulted array is necessarily less than or equal to the length of the original array.

Result for filter(fn int (x:int) ⇒ not (x==0), [0,1,0,0,2,0,3]): [1, 2, 3]

reduce (existing)

Type: ((a * a → a) * a * [a]) → a

Result for reduce(f, x, []): x

Result for reduce(f, x, [y0]): f(x, y0)

Result for reduce(f, x, [y0, y1]): f(f(x, y0), y1)

And so on.

You may have noticed that this seems very similar to the F# function List.fold. List.fold is actually a more general function, meaning it can do more than reduce.

scan (yours!)

Type: ((a * a → a) * a * [a]) → [a]

Result for scan(f, x, []): [x]

Result for reduce(f, x, [y0]): [x, f(x, y0)]

Result for reduce(f, x, [y0, y1]): [x, f(x, y0), f(f(x, y0), y1)]

And so on.

Testing your solution

We provide several tests in the handed out code. Once your solution passes all the relevant tests, great! But please write your own test programs as well! One can always write more tests.

The grand guide to implementing iota

The handed out compiler has an implementation of iota. This section describes that implementation.

First, let’s consider which parts of the compiler will need to be modified. iota isn’t just a new operator, a group of new literals or an improved optimization; it is an entirely new language construct. There is no way to implement something similar to iota in a Fasto program with existing constructs only. Think about that.

These are the files that we will need to modify:

  • Lexer.fsl

  • Parser.fsp

  • TypeChecker.fs

  • Interpreter.fs

  • CodeGen.fs

It’s the same files as in task 1. The difference is that we’ll write much more code.

Here is a file that you specifically should never modify:

  • AbSyn.fs

This is because we have been kind enough to define all relevant constructors for you. From AbSyn.fs:

type Exp<'T> =
  (* Highlevel implementations *)

  (* ... *)

  | Iota      of Exp<'T> * Position
  | Map       of FunArg<'T> * Exp<'T> * 'T * 'T * Position
  | Reduce    of FunArg<'T> * Exp<'T> * Exp<'T> * 'T * Position
  | Scan      of FunArg<'T> * Exp<'T> * Exp<'T> * 'T * Position
  | Replicate of Exp<'T> * Exp<'T> * Position

  (* ... *)

This is already done. It might seem like filter and scan are much more complex to implement than replicate, since they have all these extra arguments, but it’s mostly in the parser stage that this is very true, and we’ll get to that.

Step 1: Extending the lexer

We must tell the compiler to handle iota in our very own way. In Lexer.fsl, we add

       | "iota"         -> Parser.IOTA pos

to the keyword (s, pos) function. Then, in Parser.fsp, we add

%token <(int*int)> IOTA

to the top section. This last part is necessary because lexeme names are stored in the parser (for no particular reason).

Step 2: Extending the parser

The compiler now reads iota and turns it into Parser.IOTA, but no rule in the parser uses that, so the parser will give up on any program containing iota.

iota(n) is an expression, so we find the Exp : group of rules in Parser.fsp and add this:

        | IOTA LPAR Exp RPAR
                        { Iota ($3, $1) }

At this point we can successfully parse iota into the abstract syntax tree, but there’s no code capable of generating MIPS code for the construct, nor any type checker or interpreter.

Step 3: Checking the types

Before Fasto generates code, it does type checking to make sure that the input program makes sense. In TypeChecker.fs, we find this code:

    | Iota (n_exp, pos) ->
        let  (e_type, n_exp_dec) = checkExp ftab vtab n_exp
        if e_type = Int
        then (Array Int, Iota (n_exp_dec, pos))
        else raise (MyError ("Iota: wrong argument type "+ppType e_type, pos))

This piece of code checks that the argument n in iota(n) is an integer. First it needs to find the type of n and also check that that type checks out — it is important that you understand that n can be any expression of type int, and not just a plain integer. For example, n could be 2 + 5, which is fine, or it could be 2 + true, which isn’t okay, and which the checkExp call in the above code would throw an error for.

If everything checks out, (Array Int, Iota (n_exp_dec, pos)) is returned. Here, Array Int is our internal representation of the type [int], and Iota (n_exp_dec, pos) is an expression that has the same inner expression as the original Iota (n_exp, pos), but type checked and decorated with a type (see UntypedExp and TypedExp in AbSyn.fs).

Step 4: Interpreting

Before we get started on the code generator, we’ll take a detour in interpreter land. Typically (and certainly in this case), filling in the interpreter code is much easier than filling in the code generator code. This is because we don’t have to think about low-level MIPS details, and because we have the entire F# language and system library at our fingertips.

In Interpreter.fs, we find this code:

let rec evalExp (e : UntypedExp, vtab : VarTable, ftab : FunTable) : Value =
  match e with
  (* ... *)
  | Iota (e, pos) ->
        let sz = evalExp(e, vtab, ftab)
        match sz with
          | IntVal size ->
              if size >= 0
              then ArrayVal( List.map (fun x -> IntVal x) [0..size-1], Int )
              else let msg = sprintf "Error: In iota call, size is negative: %i" size
                   raise (MyError(msg, pos))
          | _ -> raise (MyError("Iota argument is not a number: "+ppVal 0 sz, pos))

This piece of code first evaluates the expression in iota’s single argument. If that is negative, it raises an error, since we cannot have an array with a negative length. Otherwise, if everything’s good, we make good use of F#'s list comprehension feature `[0..size-1] to generate a list of the range values.

evalExp outputs a value of type, well, Value, which you can find in AbSyn.fs. Our Iota case returns a Value value using the ArrayVal constructor, because iota generates an array.

We can now run bin/fasto -i tests/iota.fo, which should print 0 1 2 3 4 5 6 (separated by newline). If this happens, you have successfully interpreted a Fasto program with iota in it!

Step 5: Generating the MIPS code

Now for the fun part. In CodeGen.fs, we need to extend the compileExp function to also generate a list of MIPS instructions for the Iota case:

  | Iota (n_exp, (line, _)) ->

We do the following:

  • 1. Check that n_exp isn’t negative (like it’s also checked the interpreter).

  • 2. Allocate space for the return array, and put the new array in the designated output variable place, which is defined in the beginning of the function:

let rec compileExp  (e      : TypedExp)
                    (vtable : VarTable)
                    (place  : Mips.reg)
                  : Mips.Instruction list =

  • 3. Fill the allocated space with numbers 0, …​, n - 1.

  • 4. Loop over each index in the loop, and set the value at that index.

You can do point 1 by creatively using MIPS instructions.

Point 2 can be done by calling dynalloc (size_reg, place, Int) dynalloc is a function in the handed out code that can allocate (but not free) space on a small heap that is built into every MIPS assembly file; see the code for detailing comments.

Point 3 can be achieved by making an iterator register, initializing it to 0, and adding a 1 to it until it reaches the the value of n_exp.

For point 4 you need to a label and a conditional branch instruction of some sort, so that you can build a loop.

Here is a full, heavy chunk of code (with comments) which does all that:

  | Iota (n_exp, (line, _)) ->
      let size_reg = newName "size_reg"
      let n_code = compileExp n_exp vtable size_reg
      (* size_reg is now the integer n. *)

      (* Check that array size N >= 0:
         if N - 1 >= 0 then jumpto safe_lab
         jumpto "_IllegalArrSizeError_"
         safe_lab: ...
      *)
      let safe_lab = newName "safe_lab"
      let checksize = [ Mips.ADDI (size_reg, size_reg, "-1")
                      ; Mips.BGEZ (size_reg, safe_lab)
                      ; Mips.LI ("5", makeConst line)
                      ; Mips.J "_IllegalArrSizeError_"
                      ; Mips.LABEL (safe_lab)
                      ; Mips.ADDI (size_reg, size_reg, "1")
                      ]

      let addr_reg = newName "addr_reg"
      let i_reg = newName "i_reg"
      let init_regs = [ Mips.ADDI (addr_reg, place, "4")
                      ; Mips.MOVE (i_reg, "0") ]
      (* addr_reg is now the position of the first array element. *)

      (* Run a loop.  Keep jumping back to loop_beg until it is not the
         case that i_reg < size_reg, and then jump to loop_end. *)
      let loop_beg = newName "loop_beg"
      let loop_end = newName "loop_end"
      let tmp_reg = newName "tmp_reg"
      let loop_header = [ Mips.LABEL (loop_beg)
                        ; Mips.SUB (tmp_reg, i_reg, size_reg)
                        ; Mips.BGEZ (tmp_reg, loop_end)
                        ]
      (* iota is just 'arr[i] = i'.  arr[i] is addr_reg. *)
      let loop_iota   = [ Mips.SW (i_reg, addr_reg, "0") ]
      let loop_footer = [ Mips.ADDI (addr_reg, addr_reg, "4")
                        ; Mips.ADDI (i_reg, i_reg, "1")
                        ; Mips.J loop_beg
                        ; Mips.LABEL loop_end
                        ]
      n_code
       @ checksize
       @ dynalloc (size_reg, place, Int)
       @ init_regs
       @ loop_header
       @ loop_iota
       @ loop_footer

We can now compile the iota.fo test and run the resulting iota.asm file with the MARS MIPS emulator:

bin/fasto -c tests/iota.fo
java -jar lib/Mars4_5.jar tests/iota.asm

This will print the same as the interpreter.

For less typing, you can instead run our script:

Implementing replicate, filter and scan

To implement replicate, filter and scan in Fasto, you need a good understanding of:

  1. how to extend the lexer, parser, type checker, code generator, and interpreter in general;

  2. how to generate MIPS code containing loops in particular;

  3. how to parse both function names, anonymous functions and operators for use as the first arguments in filter and scan (though not replicate);

  4. how to call a function with arguments in the code generator; and

  5. how to handle different types in arrays.

Task 1 should’ve prepared you for point 1, and the previous section have hopefully helped you with point 2. The places where you’re going to modify code are the same as with iota.

To get help with point 3, 4 and 5, read on.

Parsing functions and operators

Both filter and scan take a function as the first argument: filter takes a function of one argument (like map), while scan takes a function of two arguments (like reduce). Much like with map and reduce, your lexer and parser must be able to handle both kinds of functions:

  1. A function name. This should be simple to parse.

  2. An anonymous function. Recall that Fasto’s anonymous functions are of the form fn Type ( Parameters ) ⇒ Expression.

In grammar the function-argument is already defined as the FunArg nonterminal. Please use that!

%type <AbSyn.UntypedFunArg> FunArg
....
FunArg : ID    { FunName (fst $1 ) }
       | FN Type LPAR RPAR ARROW Exp
               { Lambda ($2, [], $6, $1) }
       | FN Type LPAR Params RPAR ARROW Exp
               { Lambda ($2, $4, $7, $1) }
;

The Fasto AST also has a type for this variation of functions: FunArg. Please use that. Here it is for your quick reference:

and FunArg<'T> =
    FunName of string
  | Lambda of Type * Param list * Exp<'T> * Position

Calling functions in the code generator

For both filter and scan, you must be able to call the same function for each element in an array. But in the code generator, what is a function?

Once we have ventured over to MIPS assembly land, we don’t have proper functions anymore, but instead we have a lot of labels. How do you then call a function?

We have already implemented a helper function for you, named applyFunArg, which supports both call cases, i.e., named function and anonymous function. Its signature is:

applyFunArg ( ff     : TypedFunArg
            , args   : Mips.reg list
            , vtable : VarTable
            , place  : Mips.reg
            , pos    : Position
            ) : Mips.Instruction list =

Finally, we strongly suggest that:

  • you start by writing (on a piece of paper) a loop-based C-pseudocode documenting the code you intend to generate for filter and scan.

  • then you implement your MIPS code generation based on that.

  • you take a look and understand how map and reduce are implemented before implementing filter and scan, respectively---because they are very similar.

  • you remember that for filter you may over-allocate a result array of same length as the original, but you count up how many element succeed under the predicate argument and set the logical size of the result array at the very end.

Handling different array types

Different types have different value sizes:

  • A char takes up a single byte.

  • A byte also takes up a single byte.

  • An int takes up 4 bytes, i.e. a MIPS32 word.

  • An array type takes up a number of bytes equal to the length of the array times the length of a single element in the array.

In replicate, filter and scan, you need to use different store and load instructions for char/bool and int, since they take up different amounts of space. For char/bool, you can use SB and LB (store byte and load byte). For int, you can use SW and LW (store word and load word).

You need storing and loading since both filter and scan writes to and reads from arrays.

It is a good idea to start with a simplified approach, where e.g. you only support int arrays in your replicate, filter, and scan. Once you get that working, it shouldn’t be too hard to support also bool and char arrays.

The code generator code for iota uses SW (but not LW, since it doesn’t read from anywhere, but generates all its output by itself), so use that for inspiration.
